# EvoSQL 严格实现重构计划

## 目标
将当前“接近 EvoSQL 的最小可运行版”重构为更严格符合论文流程的实现，重点补齐以下能力：

1. 严格的初始上下文构建 `C^(0)`
2. 多轮候选采样与上下文演化
3. 严格区分 `SCR` 与 `ECA` 两种 refinement 流程
4. 自适应调度与更合理的 early-stop
5. 最终候选池上的统一决策与置信度排序

---

## 一、当前问题

### 1. 初始上下文不完全符合论文
当前实现先做了一次 schema linking，再进入迭代。
论文描述更接近：
- 先构建完整初始上下文 `C^(0)`
- 再在迭代中通过 `SCR/ECA` 精炼上下文

### 2. SCR 只做了表级精炼
当前 `SCRRefiner` 只保留候选 SQL 引用到的表，没有保留列级上下文，也没有显式重置结构提示。

### 3. ECA 锚点不完整
当前 ECA 只把代表 SQL 和 execution signature 写回上下文。
论文要求更接近：
- 保留高一致性簇
- 将高置信 SQL 及其真实执行结果作为语义锚点注入下一轮上下文

### 4. 调度与 early-stop 太粗糙
当前只根据问题关键词决定轮数和采样数，并用“表集合是否稳定”做 early-stop。
这不够支撑论文中的“自适应调度与收敛”。

### 5. 最终输出机制不严格
当前最终 SQL 主要来自：
- ECA 最大簇代表
- 或最高启发式分数候选

还没有实现：
- 最终候选池统一重排
- 多数投票 / 结果一致性决策
- 更合理的置信度分解

---

## 二、重构原则

1. 不再混入 mock 逻辑
2. 保持当前前后端接口尽量兼容
3. 优先重构后端 EvoSQL 主流程
4. 每一步都保留可解释性输出，便于答辩展示
5. 先做“结构正确”，再优化“效果更好”

---

## 三、分阶段实现计划

## 阶段 A：上下文结构重构
目标：严格显式化论文中的上下文对象 `C^(t)`

### 要做的事
1. 扩展 `ContextState`
   - 明确拆分：
     - `instruction`
     - `question`
     - `schema_subset`
     - `history_hints`
     - `external_knowledge`
     - `semantic_anchors`
     - `iteration`
2. 显式构造 `C^(0)`
   - 不再把 schema linking 直接等价为初始上下文
   - 先保留完整 schema + 外部知识 + 问题
3. 将 schema linking 改为 `C^(0) -> C^(1)` 的预处理或第一轮 refinement
4. 在 trace 中记录每一轮上下文变化

### 产出
- 统一的上下文演化数据结构
- 前端可展示每轮上下文摘要

---

## 阶段 B：重写 SCR 流程
目标：让 `SCR` 严格符合“解析-聚合-重置提示”

### 要做的事
1. 从候选池中提取：
   - 引用表
   - 引用列
2. 对表列做 union 聚合
3. 按聚合结果裁剪 schema
   - 不只是表级裁剪
   - 还要支持列级裁剪
4. 显式重置结构提示
   - `history_hints = []`
   - `semantic_anchors = []`
5. 输出新的 `C^(t+1)`

### 产出
- 列级 SCR 上下文精炼
- trace 中可见：
  - 保留了哪些表
  - 保留了哪些列
  - 清空了哪些提示

---

## 阶段 C：重写 ECA 流程
目标：让 `ECA` 严格符合“执行-聚类-保留 Top-k-锚点构建”

### 要做的事
1. 对候选 SQL 全部执行
2. 依据执行结果构造更稳定的 result signature
   - 行数
   - 列集合
   - 前几行摘要
   - 聚合结果特征
3. 按结果一致性聚类
4. 选出 Top-k 高一致性簇
5. 从高一致性簇中提取语义锚点：
   - SQL 文本
   - 执行结果摘要
   - 命中表列
6. 将这些锚点注入下一轮上下文

### 产出
- 完整 ECA 语义锚点机制
- 前端可展示：
  - 哪些候选进入同一簇
  - 哪些簇被保留
  - 哪些锚点进入下一轮

---

## 阶段 D：重写调度器与 early-stop
目标：实现更合理的自适应调度和收敛判断

### 要做的事
1. 调度输入增加：
   - 问题复杂度
   - schema 规模
   - 候选执行成功率
   - 候选分歧程度
   - 空结果比例
2. 动态决定：
   - 最大轮数
   - 每轮采样数
   - ECA top-k
3. early-stop 不只看表集合
   - 还看候选池是否稳定
   - 还看簇结构是否稳定
   - 还看最终高分候选是否稳定
4. 支持“失败提前终止”
   - 当候选长期全部无效时直接结束

### 产出
- 真正的 adaptive scheduler
- 更符合论文的收敛控制

---

## 阶段 E：重写 score 与最终选择
目标：让“置信度最高的最优 SQL”有明确依据

### 要做的事
1. 将候选分数拆成多因子：
   - `prior_score`
   - `schema_score`
   - `structure_score`
   - `execution_score`
   - `consistency_score`
   - `final_score`
2. 结构分判断：
   - 是否满足聚合需求
   - 是否满足 top-k
   - 是否满足趋势问题时间分组
   - 是否满足 join 路径合理性
3. 执行分判断：
   - 是否成功执行
   - 是否空结果
   - 是否与主簇一致
4. 最终选择策略
   - 先按簇一致性筛
   - 再按 `final_score` 排
   - 必要时做多数投票
5. 输出详细 score breakdown

### 产出
- 可解释候选打分
- 最终 SQL 的选择依据可直接展示

---

## 阶段 F：前端可解释性同步升级
目标：让论文流程能在页面中被清楚展示

### 要做的事
1. 增加每轮上下文卡片
   - schema subset
   - semantic anchors
   - history hints
2. 增加簇视图
   - 每轮候选如何分簇
3. 增加 score breakdown 展示
4. 增加最终决策说明
   - 为什么选这条 SQL
   - 为什么淘汰其他候选

### 产出
- 可直接用于答辩演示的 EvoSQL 过程面板

---

## 四、推荐实施顺序

1. 阶段 A：上下文结构重构
2. 阶段 B：SCR 重写
3. 阶段 C：ECA 重写
4. 阶段 D：调度器与 early-stop 重写
5. 阶段 E：score 与最终选择重写
6. 阶段 F：前端过程可视化升级

---

## 五、每阶段验收标准

### 阶段 A 验收
- 能明确看到 `C^(0)` 和 `C^(t)` 的结构变化

### 阶段 B 验收
- SCR 后 schema 真正缩减到表列级
- 提示被显式清空

### 阶段 C 验收
- ECA 能输出簇与锚点
- 下一轮上下文能读取这些锚点

### 阶段 D 验收
- 轮数、采样数、top-k 不再是固定启发式
- early-stop 由多因素触发

### 阶段 E 验收
- 每个候选都有结构化 score breakdown
- 最终 SQL 选择有清晰依据

### 阶段 F 验收
- 前端能完整展示论文流程
- 可用于答辩演示

---
